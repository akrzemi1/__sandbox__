3.2. Mathematical soundness

In this section we respond to the claim that the requirement on CommonReference is required to make concept StrictWeakOrdering "mathematically sound".

The notion of being "mathematically sound" has no formal definition and is subject to personal judgement or aesthetic preference. In order to facilitate the discussion we would need to define it. One possible attempt is the following. Because in mathematics, the concept of strict weak ordering is described for elements a, b, c from one domain D, express the corresponding requirements in C++ for objects a, b, c of the same type D (which is far from obvious a step: in mathematics a domain can be composed of values represented by C++ object from two different types). In order to do this, one has to oblidge the user to provide a number of additional definitions (however difficult, impractical or error prone) to create a single type D and express the axioms token-by-token as they are spelled out in mathematics for objects of type D. We do not propose this, and instead offer the following alternative. Provide a formal system based on the mathematical concept of strict weak ordering that preserves its essential properties and gives us the benefits of the mathematical concept, but works for two types T and U without requiring a common type and corresponding conversions.

What do we require of the syntactic and semantic constraints on concepts:

  • That they are sufficient to describe preconditions and postconditions on functions dealing with order.
  • That they are sufficient for the run-time injected diagnostic code (such as inside assert()) to be able to test if all the semantic constraints were met.

We believe the semantic constraints expressed with the following functions meet the above requirements: 

template <typename A>
bool irreflexivity(const A& a)
{
  return !comp(a, a);
}

template <typename A, typename B>
bool asymmetry(const A& a, const B& b)
{
  if (comp(a, b))
    return !comp(b, a);
  return true;     
}

template <typename A, typename B, typename C>
bool transitivity(const A& a, const B& b, const C& c)
{
  if (comp(a, b))
    if (comp(b, c))
      return comp(a, c);
  return true;     
}

template <typename A, typename B, typename C>
bool transitivity_of_equivalence(const A& a, const B& b, const C& c)
{
  if (!comp(a, b) && !comp(b, a))
    if (!comp(b, c) && !comp(c, b))
      return !comp(a, c) && !comp(c, a);
  return true;     
}

Where each A, B, C can be either T or U. These require that the predicate apart from working for comp(t, u) and comp(u, t) also needs to work for comp(t, t) and comp(u, u), but require no existence of a common type or conversions to this common type.
